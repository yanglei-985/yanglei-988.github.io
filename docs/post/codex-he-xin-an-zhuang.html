Codex 配置指南：把终端变成你的第二大脑

如果说浏览器是信息入口，Obsidian 是知识仓库，那么 Codex 就是你放在终端里的“行动层”。它不只是回答问题，而是能读仓库、改文件、跑测试、提交代码，甚至帮你把一个 GitHub 链接变成一篇真正上线的文章。

这篇指南写给已经开始用 AI 做事的人：你不需要先成为程序员，但你需要把工具配置成“可持续工作的系统”。

先给结论

Codex 配置的核心不是装一个命令，而是完成四件事：

  1. 让 Codex 能被终端调用：安装 CLI。
  2. 让 Codex 能访问模型：配置登录或 API Key。
  3. 让 Codex 知道怎么做事：写好全局配置和项目规则。
  4. 让 Codex 安全地动手：设置权限、沙盒、Git 工作流。

配置好以后，你的工作方式会从：

我打开文件 → 我复制代码 → 我问 AI → 我手动改 → 我测试

变成：

我给目标 → Codex 看仓库 → Codex 修改 → Codex 验证 → 我审阅和推送

一、安装前准备

先确认你的系统里有这些基础工具。

node -v
npm -v
git --version
python3 --version

如果这些命令大部分都能输出版本号，说明环境已经差不多了。

建议你再确认当前目录是不是你的工作目录，比如：

pwd
ls

我的习惯是把项目统一放在一个固定文件夹里，例如：

~/桌面/宠物/网站

这样以后你只要说“去我的网站仓库改一下”，Codex 就知道从哪里开始。

二、安装 Codex CLI

常见方式是通过 npm 安装：

npm install -g @openai/codex

安装完成后检查：

codex --version
codex --help

如果提示 command not found，通常是 npm 的全局 bin 目录没有加入 PATH。可以先查看：

npm bin -g

然后把输出路径加入你的 shell 配置，例如 ~/.bashrc 或 ~/.zshrc。

export PATH="$PATH:$(npm bin -g)"

保存后重新加载：

source ~/.bashrc

三、配置登录或 API Key

Codex 要真正工作，必须能访问模型。一般有两种方式。

方式 A：浏览器登录

codex login

它会打开浏览器，让你用 OpenAI 账号授权。适合个人电脑，最省心。

方式 B：环境变量 API Key

export OPENAI_API_KEY="你的_api_key"

如果想长期生效，可以写进 ~/.bashrc：

echo 'export OPENAI_API_KEY="你的_api_key"' >> ~/.bashrc
source ~/.bashrc

注意：不要把 API Key 写进 GitHub 仓库，不要提交到 git，不要发给别人。

如果你不确定是否配置成功，可以用：

env | grep OPENAI

只确认变量存在即可，不要把完整 key 截图给别人。

四、写全局配置文件

Codex 通常会读取用户目录下的配置文件。你可以先创建配置目录：

mkdir -p ~/.codex
nano ~/.codex/config.toml

一个稳妥的入门配置可以这样写：

model = "按你的账号可用模型填写"
approval_policy = "on-request"
sandbox_mode = "workspace-write"

[workspace]
trust = true

这里最重要的是两个字段：

  • approval_policy = "on-request"：遇到高风险操作先问你。
  • sandbox_mode = "workspace-write"：允许它修改当前工作区，但不要随便动系统其他地方。

如果你只是写文章、改网页、修脚本，这个配置已经够用了。

五、权限策略：别一上来就开危险模式

Codex 最强的地方是“能动手”，但这也是最需要边界的地方。

我建议按这个顺序使用：

  1. 只读模式：让它先看仓库、解释结构、找问题。
  2. 工作区写入：允许它改当前项目文件。
  3. 需要确认的命令：比如安装依赖、联网、push。
  4. 危险模式：只在你完全知道后果时使用。

尤其不要让它无确认执行这些事情：

rm -rf
sudo
删除数据库
覆盖远程分支
泄露 token

好的 AI 工作流不是“完全放权”，而是“让它做重复劳动，让你做最终判断”。

六、为每个项目写 AGENTS.md

如果全局配置是“Codex 怎么运行”，那么 AGENTS.md 就是“这个项目怎么做事”。

在你的仓库根目录创建：

touch AGENTS.md

可以写成这样：

# AGENTS.md

## 项目目标
这是我的个人博客仓库，使用 Gmeek，通过 GitHub Issues 生成静态页面。

## 工作规则
- 修改前先 git pull。
- 不直接改 docs 生成物，除非明确要求。
- 文章源头通常是 GitHub Issue 正文。
- 修改脚本后要运行基本校验。
- 每次提交前展示 git diff。

## 安全规则
- 不提交 token、cookie、密码。
- 不删除重要文件。
- push 前需要用户确认。

这样你每次打开项目，Codex 都能先读这份说明，不会把你的博客逻辑搞反。

七、把 GitHub 仓库变成可协作工作区

以后你给 Codex 一个 GitHub 链接，它可以这样做：

git clone https://github.com/你的用户名/你的仓库.git
cd 你的仓库
git status

如果仓库已经存在，就不要重复 clone，而是：

cd 你的仓库
git pull --ff-only
git status --short --branch

这是非常重要的一步。因为你可能在网页上改了文件，GitHub Action 也可能自动提交了生成结果。如果 Codex 不先 pull，就可能在旧版本上工作。

我现在更喜欢的节奏是：

同步仓库 → 阅读上下文 → 修改文件 → 本地验证 → commit → push

这比“直接在网页上乱改”稳定得多。

八、让 Codex 知道你的长期上下文

很多人用 AI 的痛点不是模型不聪明，而是每次都要重新解释背景。

解决办法很简单：给仓库放一个上下文文件，例如：

context.md

里面写清楚：

  • 这个项目是什么。
  • 内容源头在哪里。
  • 哪些文件不能直接改。
  • 发布流程是什么。
  • 已经踩过哪些坑。
  • 当前待办是什么。

比如 Gmeek 博客就应该写明：

文章源头是 GitHub Issue，不是 docs。
git push 不触发构建，Issue 编辑才触发。
复杂 HTML 要注意 GitHub Markdown API 的清洗。

这样下一个 AI 或下一次 Codex 会话，就能从 context.md 继续，而不是从零开始。

九、一个推荐的日常提示词

你可以这样对 Codex 说：

先 git pull 同步我的仓库，然后阅读 context.md 和 AGENTS.md。
接着帮我完成这个任务：……
修改前先说明计划，修改后运行必要验证，最后给我 git diff 和 commit message。

如果是博客文章任务，可以说得更具体：

请按照 context.md 的 Gmeek 发布规则，把这篇文章整理成可直接粘贴到 GitHub Issue 的 Markdown。
要求保留宝玉风格的 section 卡片排版，检查 section 标签开闭数量，输出到 drafts 文件夹。

这样的提示词不是为了“命令 AI”，而是为了降低沟通成本。你越清楚，AI 越像工具；你越模糊，AI 越像聊天机器人。

十、推荐的提交工作流

当 Codex 修改完文件，不要急着 push。先看状态：

git status --short --branch

再看具体改了什么：

git diff

确认没有问题后提交：

git add 需要提交的文件
git commit -m "写清楚这次改了什么"

最后再 push：

git push origin main

如果终端要求输入 GitHub 用户名：

Username: 你的 GitHub 用户名
Password: 粘贴 Personal Access Token，不是账号密码

GitHub 现在不支持用登录密码 push，必须用 token 或 SSH key。

十一、常见问题

1. Codex 能不能直接改 GitHub 仓库？

可以，但本质是通过 git：clone、修改、commit、push。能不能 push 取决于你本地有没有 GitHub 权限。

2. Codex 会不会改坏文件？

会有可能。所以要让它先 git diff，再 commit。Git 的意义就是给你后悔药。

3. 我需要每个项目都配置一次吗？

全局配置只要一次。每个项目建议放自己的 AGENTS.md 和 context.md。

4. 它能不能一直看着我的仓库？

可以定期 git fetch 或 git pull 检查变化。但通常更推荐在每次任务开始时主动同步，这样更可控。

最后一句话

Codex 不是“另一个聊天窗口”，它更像一个坐在你电脑里的执行者。

你真正要配置的，也不只是模型名、API Key、命令行参数，而是一套工作秩序：

上下文清楚 → 权限克制 → 修改可追踪 → 验证可复现 → 发布可回滚

当这套秩序建立起来，AI 就不再只是帮你写几段文字，而是开始参与你的真实工作流。
